> 분석/리팩토링: `forest_reco` 패키지 + 모바일 앱으로 재구성

## 1. 이 파일은 무엇인가 (한눈에)

강원권을 대상으로 한 **위치 기반 산림 분석 Colab 노트북**입니다. 휴대폰으로 산림
사진을 찍거나 GPS 좌표를 입력하면, 그 위치의 **임상도(숲의 종류 지도)** 와 **DEM
(고도 지도)** 를 조회해 임분 정보를 보여주고, 머신러닝 분류기로 "관리 우선순위 /
생육 적합도 / 병해충 취약도"를 예측한 뒤 Gemini로 자연어 리포트를 생성합니다.
마지막엔 Streamlit + cloudflared 로 모바일 웹앱 형태의 데모를 띄웁니다.

- **셀 수**: 121개 (전부 code 셀). 같은 작업을 반복 시도한 흔적이 많아 매우 산만함
  (streamlit 실행/종료/터널 셀이 수십 개 중복).
- **언어/환경**: Python 3 (Colab), 데이터는 Google Drive `/content/drive/MyDrive/Forest_AI/`.

## 2. 입력 데이터 (인풋이 무엇인가)

| 데이터 | 형식 | 좌표계 | 역할 |
|---|---|---|---|
| **임상도** `51_1.shp`, `51_2.shp` | **벡터 폴리곤** (Shapefile) | EPSG:5179 (UTM-K) | 위치의 임분 속성 |
| **DEM** `gangwon_dem.tif` (여러 타일 merge) | **래스터** (GeoTIFF) | EPSG:4326 → 일부 셀에서 5179 재투영 | 고도/경사/향 |
| 사진(EXIF) / GPS / 수동 좌표 | 점(point), WGS84 | EPSG:4326 | 질의 위치 |

임상도 속성 컬럼(국가 수치임상도 표준):

| 필드 | 한글 | 의미 | 값 도메인 |
|---|---|---|---|
| `FRTP_NM` | 임종/임상 | 침엽수림/활엽수림/혼효림 (또는 인공림/천연림) | 침엽수림·활엽수림·혼효림·죽림 등 |
| `KOFTR_NM` | 수종 | 우점 수종명 | 소나무·잣나무·낙엽송·신갈나무·기타활엽수… |
| `DMCLS_NM` | 경급 | 흉고직경(DBH) 등급 | 치수(<6cm)·소경목(6–18)·중경목(18–30)·대경목(≥30cm) |
| `AGCLS_NM` | 영급 | 10년 단위 수령 등급 | 1영급(1–10년)…6영급(51년+) |
| `DNST_NM` | 밀도 | 수관 울폐도 | 소(≤50%)·중(51–70%)·밀(≥71%) |

> ⚠️ **사용자 우려에 대한 정답**: "shp가 래스터라 점이 격자에 들어가는지 계산하는
> 함수가 필요한가?" → **임상도(.shp)는 래스터가 아니라 벡터 폴리곤입니다.** 점이
> 어느 폴리곤에 들어가는지는 *point-in-polygon*(`contains`/공간조인)으로 풀며, 원본
> 노트북도 이미 그렇게 하고 있었습니다. 따라서 임상도에는 별도 격자 계산이 필요
> 없습니다. **격자(셀) 계산이 필요한 쪽은 DEM(.tif) 래스터**이고, 그것은
> `dataset.index(x, y) → (row, col)` 로 "점이 떨어진 셀"을 찾는 것이 정답입니다.
> 다만 원본은 이 부분에 치명적 버그가 있었습니다(§4-④). 본 리팩토링은 두 경로를
> 명확히 분리해 구현했습니다 → [`forest_reco/geo.py`](forest_reco/geo.py),
> [`forest_reco/forest_map.py`](forest_reco/forest_map.py).

## 3. 모델 구조 (모델이 어떻게 되어있나)

노트북에는 사실상 **세 가지 산출물**이 있습니다.

1. **1차 모델 (CELL 3)** — `RandomForest` + `XGBoost` 가 `priority`(관리 우선순위:
   높음/보통/낮음)를 분류.
2. **2차 모델 (CELL 5–7)** —
   - `growth_model`(XGBoost) → 생육 적합도
   - `pest_model`(RandomForest) → 병해충 취약도
   - `make_management_priority`(규칙)로 둘을 합쳐 최종 관리 우선순위.
3. **Gemini 리포트 (CELL 12)** — 위 결과를 프롬프트에 넣어 자연어 진단문 생성.

### ❗ 모델의 근본 결함: 순환 학습(데이터 누수)
`priority`, `growth_suitability`, `pest_vulnerability` 라벨은 **사람이 손으로 짠
if/else 규칙(`make_*`)이 입력 피처로부터 결정론적으로 생성**합니다. 그런 다음 **정확히
같은 피처**로 RandomForest/XGBoost를 학습합니다. 즉 모델은 자기 입력으로 만든 규칙을
그대로 다시 외우는 것이라, 보고되는 정확도(거의 100%)는 **"규칙 암기" 점수일 뿐
예측력·일반화와 무관**합니다. (이 위에 무작위 `train_test_split`은 인접 폴리곤
누수까지 더합니다.)

→ 결론: 이 용도라면 ML은 불필요한 우회입니다. **규칙이면 규칙엔진으로 투명하게**
쓰는 편이 정확하고 설명 가능하며, 진짜 ML을 원한다면 라벨을 **외생 관측치**(실제
피해목 조사·생장 실측)에서 가져오고 공간 블록 교차검증을 써야 합니다.

## 4. 발견된 결함 (맹점) 과 본 리팩토링의 해결

`docs/research/critique.json`의 적대적 코드 감사(16건)와 자체 분석을 종합했습니다.

| # | 심각도 | 결함 | 본 프로젝트의 해결 |
|---|---|---|---|
| ① | 치명 | **순환학습/데이터누수** — 규칙으로 만든 라벨을 같은 피처로 ML 재학습 | ML 제거. 투명한 **적지적수 점수 엔진**으로 재설계 → [`recommender.py`](forest_reco/recommender.py) |
| ② | 치명 | **제품 목표 불일치** — 목표는 "심을 수종 추천"인데 구현은 "기존 임분 진단"만 | **식재 적합 수종 Top-N 추천**을 신규 구현(핵심 기능) |
| ③ | 치명 | `aspect_to_direction()`/`aspect_dir` **미정의 → NameError/KeyError 크래시** | `geo.aspect_to_direction()` 정의 + 파이프라인에서 생성 |
| ④ | 치명 | **좌표계 혼용** — 5179 DEM의 `dem.index(lon,lat)`에 원시 위경도 전달 | `geo.sample_raster()`가 점을 **항상 래스터 CRS로 변환** 후 인덱싱 |
| ⑤ | 높음 | **경계검사 부재 → 음수 인덱스 silent wrap** (엉뚱한 고도를 정상값처럼 반환) | bounds 검사 + nodata/NaN 검사 → 범위 밖이면 명시적 예외 |
| ⑥ | 높음 | **4326 격자(도 단위)에서 경사 계산** → 경사 왜곡 | 경사/향은 **미터 단위(5179)** 에서만 계산. 4326 DEM은 자동 재투영 |
| ⑦ | 높음 | **공간 인덱스 없음** → 매 호출 전수 스캔 + shapefile 매번 재로딩 | `sindex` 기반 질의 + `@st.cache_resource` 로 1회 로딩 |
| ⑧ | 높음 | **bare `except`** 가 좌표계 버그를 NaN으로 은폐 | 좁은 예외(`RasterSampleError`)와 명시적 사유 반환 |
| ⑨ | 높음 | CELL 13/96 **들여쓰기 SyntaxError** | 모듈화로 셀 순서 의존 제거(정적 검사 통과) |
| ⑩ | 중간 | **전역 가변 상태**(dem/slope/encoders…)로 셀 순서마다 다른 결과 | 순수 함수 + 의존성 주입(`DataSources`, `SiteContext`) |
| ⑪ | 중간 | `'5' in str(영급)` 부분일치 → `'15영급'` 오분류 위험 | 도메인 정규화 + 명시 매핑(부분일치 제거) |
| ⑫ | 중간 | centroid가 도넛형 폴리곤 밖에 떨어질 수 있음 | (권장) `representative_point()` / zonal stats 사용 명시 |
| ⑬ | 중간 | Gemini가 누수 결과를 "전문가 진단"으로 과신 서술 | LLM은 **설명만**, 추천 결정은 결정론적 엔진. 면책·근거 명시 |
| ⑭ | 중간 | `forest.crs` 미지정 시 `to_crs` 실패 | 로드시 CRS 검증·폴백(`.prj` 우선, 5179 폴백) |

> **좌표계 핵심 교훈**: 임상도(EPSG:5179)와 DEM(파일별 상이)을 단일 기준으로 통일하고,
> **모든 점 질의는 입력 WGS84 → 대상 데이터 CRS로 변환 후** 수행. 임상도 CRS는
> 배포본에 따라 **5181**일 수도 있으므로 코드가 `.prj`에서 읽도록 했습니다.

## 5. 사용법 (원본 vs 신규)

### 원본 노트북 흐름
1. Drive 마운트 → `51_*.shp`, `*.tif` 적재 → DEM merge.
2. `forest.contains(point)`로 임분 조회, `dem.index`로 고도.
3. 규칙으로 라벨 생성 → ML 학습 → `joblib` 저장.
4. Gemini 키 입력 → `forest_ai_report()` → LLM 리포트.
5. Streamlit 앱(`%%writefile app.py`) → cloudflared 터널.

### 신규 (이 저장소)
```bash
pip install -r requirements.txt

# 데이터 없이 즉시 데모 (합성 강원권 데이터 자동 생성)
python scripts/run_demo.py --lat 37.95 --lon 127.66 --goal 탄소흡수 --mock

# 모바일 앱
streamlit run app/streamlit_app.py
```
실데이터가 있으면 사이드바에서 폴더만 지정하면 됩니다. 자세한 내용은
[`README.md`](README.md).

```jsonc
{
  "ok": true,
  "location": {"lat": 37.95, "lon": 127.66, "source": "exif|gps|manual"},
  "site": {"elevation_m": 1232.7, "slope_deg": 2.6, "aspect_dir": "남서향",
            "climate_zone": "온대북부", "warmth_index": 59},

  "neighborhood_species": {"신갈나무": 8, "잣나무": 5, ...},   // 경험적 근거
  "recommendations": [                                        // ★ 핵심 산출물
    {"수종": "신갈나무", "학명": "Quercus mongolica", "적합점수": 92.4,
     "주요근거": ["기후대: 온대북부 적지", "고도: 1232m ∈ 적정 200~1700m"],
     "유의사항": ["병해충 유의: 참나무시들음병…"], "탄소흡수": "상"},
```

## 6.5. 신규 코드 자체 검증 (적대적 리뷰 라운드)

리팩토링 코드도 다중 에이전트 적대적 리뷰로 재검증하여, 합성데이터 테스트가
놓친 **실제 버그 6건을 추가로 발견·검증하고 모두 수정**했습니다(각 건은 코드를
직접 실행해 재현 확인).

| 버그 | 영향 | 수정 |
|---|---|---|
| `sindex.query(point, predicate="contains")` 방향 오류 | 점-폴리곤 포함이 항상 최근접 폴백으로 샘 | `predicate="within"` |
| bilinear 보간이 래스터 **가장자리 반-픽셀**에서 예외 | DEM 가장자리 띠의 유효 점이 고도 없이 탈락 | 이웃 인덱스 클램프 |
| 재투영 시 `src_nodata` 미지정 | nodata(-9999)가 유효셀과 섞여 **가짜 88° 절벽** 생성 | `src_nodata`/`dst_nodata=NaN`/`init_dest_nodata` |
| 고도 적합도 **경계 역전** | 적정범위 가장자리(0.82)가 범위 밖(≈0.9)보다 낮음 | 가장자리값에서 연속 감쇠 |
| 빈 `SpeciesDB`가 falsy → 기본 DB로 무단 대체 | 부분집합 추천 의도 무시 | `None` 명시 검사 |
| `match()` 양방향 부분일치 과매칭 | `'나무'`→소나무 등 오매칭으로 경험가점 오염 | 단방향+최장토큰 결정적 매칭 |

→ 회귀 방지 테스트를 추가해 현재 **53개 pytest 전부 통과**합니다.

## 7. 한 줄 요약

원본은 좋은 데이터(임상도+DEM+Gemini)와 아이디어를 갖췄지만, **(1) 실행 불가능한
치명 버그, (2) 의미 없는 순환학습 모델, (3) 좌표계 오류, (4) 정작 목표인 '식재 수종
추천' 기능 부재** 라는 네 가지 문제가 있었습니다. 본 리팩토링은 이를 모두 해결하고,
설명 가능한 적지적수 추천 엔진 + 모바일 앱 + 테스트로 재구성했습니다.
