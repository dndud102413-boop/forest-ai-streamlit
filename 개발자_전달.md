## 1. 왜 / 어떻게 바꿨나 (기존 대비)

### 1-1. 가장 큰 문제: 목표와 구현의 불일치
- **기존**: `forest.contains(point)`로 그 자리에 *이미 있는* 임분을 진단(임종/수종/경급…)만 함.
- **목표**: 그 자리에 *심을* 적합 수종 추천 — 기존 코드엔 추천 로직이 **아예 없었음**.
- **수정**: 입지(기후대·고도·경사·향) → 적지적수 점수로 식재 수종을 랭킹하는
  [`recommender.py`](forest_reco/recommender.py)를 신설.

### 1-2. "AI 모델"이 사실 무의미했던 문제 (순환학습)
- **기존**: `make_priority` 같은 **사람이 손으로 짠 규칙**으로 라벨을 만든 뒤, **같은 피처**로
  RandomForest/XGBoost를 학습. → 모델은 그 규칙을 그대로 외울 뿐(정확도 ≈100%는 허수).
- **수정**: (a) 추천은 **투명한 적지적수 점수**로(설명 가능), (b) ML을 쓰려면 *제대로* —
  임상도의 **실제 수종 분포(KOFTR_NM)** 를 라벨로 한 **수종분포모델(SDM)** 을 별도 구현
  ([`sdm.py`](forest_reco/sdm.py), 8장 참고).

### 1-3. 좌표계·래스터 버그 (사용자가 걱정한 부분)
- **임상도(.shp)는 래스터가 아니라 벡터 폴리곤** → 점이 어느 폴리곤에 드는지는
  point-in-polygon(`within`). 격자 계산 불필요. (원본도 이미 `contains`를 썼음)
- **격자 계산이 필요한 건 DEM(.tif) 래스터** → `dataset.index(x,y)`로 "점이 떨어진 셀"을 찾음.
  여기서 원본은 ① 원시 위경도를 재투영 래스터에 그대로 전달 ② 경계검사 없어 음수 인덱스가
  조용히 반대편 픽셀을 읽음 — 두 치명 버그.
- **수정**: [`geo.sample_raster()`](forest_reco/geo.py)가 **좌표계 변환 + 경계검사 +
  nodata + bilinear**를 모두 안전 처리. 누락됐던 `aspect_to_direction()`도 구현.

### 1-4. 자체 검증에서 추가로 잡은 6건
다중 에이전트 적대적 리뷰로 리팩토링 코드도 재검증 → 실제 버그 6건(가장자리 bilinear 예외,
재투영 nodata로 인한 가짜 88° 절벽, 고도점수 경계역전, 공간인덱스 predicate 방향, 빈 DB 대체,
수종명 과매칭)을 코드로 재현·수정하고 회귀 테스트를 추가. (상세: [ANALYSIS.md](ANALYSIS.md))

## 8. 모델 — RandomForest/XGBoost를 계속 써야 하나? 더 나은 대안은?

### 결론 (ML은 유지·기본 사용합니다)
- **RF/XGB 자체가 문제가 아니라 *학습 방식*(규칙을 베끼는 순환학습)이 문제였습니다.** 그
  잘못된 학습만 걷어내고, 라벨을 **실관측(임상도 KOFTR_NM)** 에서 가져오는 **올바른 ML =
  종분포모델(SDM)** 으로 교체했습니다.
- **추천은 'ML(SDM) + 지식기반 점수' 하이브리드**가 가장 우수합니다. ML만 쓰면 데이터가
  적을 때 약하고 설명이 안 되며, 규칙만 쓰면 데이터에서 못 배웁니다. **둘을 합치고, ML의
  영향력을 모델 신뢰도(f1)에 비례해 자동 조절**하면 — 데이터가 좋을수록 ML이 주도하고,
  부족하면 지식기반이 받쳐줘서 항상 안전합니다.

### 구현해 둔 대안: SDM ([`sdm.py`](forest_reco/sdm.py))
- `P(수종 | 고도·경사·향·위도)` 를 임상도 실분포로 학습 (sklearn **HistGradientBoosting** —
  XGBoost/LightGBM급 성능에 추가 의존성 없음). 누수 없음.
- 추천에 **기본 사용**(`use_sdm=True`, 앱 토글 ON). 영향력은 **f1에 비례 자동 가중**
  (`sdm_weight = 30 × min(1, f1/0.5)`). 결과에 `sdm_used/sdm_quality/sdm_weight_applied` 노출.
- 합성데이터 f1은 낮게(0.18) 나와 가중치가 ~10.8/100으로 자동 축소됩니다 — **목업이 수종을
  일부러 무작위로 섞기 때문**이며 정직하게 학습한다는 신호입니다. **실 임상도에서는 환경→수종
  신호가 강해 f1이 올라가고, 그만큼 ML이 추천을 더 주도**합니다.
- 최초 1회 학습(수십 초) 후 캐시. `Settings.sdm_path` 지정 시 디스크에 저장/재사용.

### 더 정석적인 모델 옵션(실데이터 확보 시)
1. **SDM 앙상블** — 이미 **구현해 둠**(기본 off). `algos=("hgb","rf")` 또는 환경변수
   `FOREST_RECO_SDM_ALGOS=hgb,rf` 로 켜면 HGB+RF 확률 평균 앙상블이 됩니다. 추후 GLM/MaxEnt를
   더해 biomod2式 다중 앙상블로 확장 가능. (앙상블의 이득은 *실데이터*에서 robust·불확실성
   추정으로 나타나며, 합성데이터에선 미미합니다 — 알고리즘보다 데이터·피처가 병목)
2. **MaxEnt** — presence-only(표본점만 있을 때) 종분포의 사실상 표준.
3. **공간 블록 교차검증**(지역 단위 GroupKFold) — 인접 폴리곤 누수를 막아 정직한 성능 평가.
4. **기후 엔벨로프(BIOCLIM)** — 미래 시나리오로 분포 이동 추정(기후변화 적응 추천).

> 권장: **SDM(데이터 주도) + 적지적수 점수(설명·콜드스타트) 하이브리드, ML 신뢰도 비례 가중**.
> LLM은 *설명만* 담당(환각 분리).

## 9. 정식 앱으로 갈 수 있나? (로드맵)

**지금도 "연결 가능"합니다.** 추천 엔진이 순수 함수라 이미 **REST API**로 노출해 뒀습니다.

```cmd
run_api.bat        :: http://localhost:8000  (문서: http://localhost:8000/docs)
```
| 엔드포인트 | 용도 |
|---|---|
| `POST /analyze` | 위·경도 JSON → 분석/추천/설명 |
| `POST /analyze/photo` | **사진 업로드(multipart)** → EXIF GPS 자동추출 → 분석 |
| `GET /meta`, `/species`, `/health` | 드롭다운·수종목록·헬스체크 |

### 단계별 추천 경로 (Windows → 모바일)
1. **지금(완료)**: Windows에서 Streamlit 앱으로 화면 검증 + REST API 가동. → *여기까지 보고 OK면*
2. **PWA(가장 빠름)**: Streamlit 앱을 HTTPS로 배포(또는 cloudflared) → 휴대폰 브라우저에서
   "홈 화면에 추가" → 네이티브처럼 사용. 추가 개발 거의 없음.
3. **정식 네이티브 앱**: Flutter/React Native 프런트 + 위 FastAPI 백엔드.
   카메라/EXIF/GPS는 디바이스 SDK로, 분석은 API 호출. 백엔드는 그대로 재사용.

→ 즉 **"윈도우에서 먼저 보고 → 괜찮으면 모바일"** 전략이 그대로 가능합니다.
   엔진/ API는 모바일로 넘어가도 **재작성 없이 재사용**됩니다.

## 10. 개발자에게 보낼 메시지 (초안 — 복사해서 쓰세요)

> 안녕하세요. 보내주신 산림 AI 노트북을 분석·리팩토링했습니다. 요약하면,
>
> 1. **목표였던 "위치 기반 식재 수종 추천"을 새로 구현**했습니다. 기존 코드는 그 자리에 이미 있는
>    숲을 *진단*만 했고 추천 기능이 없었어요. 입지(기후대·고도·경사·향)로 적합 수종 Top-N을
>    근거와 함께 추천합니다.
> 2. **기존 ML(RandomForest/XGBoost)이 사실 의미가 없었던 부분**을 바로잡았습니다. 손으로 짠 규칙으로
>    만든 라벨을 같은 데이터로 다시 학습하는 구조라 정확도가 허수였거든요. 추천은 *설명 가능한 점수*로
>    바꾸고, ML이 필요하면 임상도 실분포로 학습하는 **종분포모델(SDM)** 을 따로 넣어 뒀습니다.
> 3. **좌표계·래스터 버그와 실행 불가 버그 다수**를 고쳤습니다(사진 위치 추출, DEM 셀 샘플링 안전화 등).
> 4. **갤럭시처럼 위치 켜고 찍은 사진을 올리면 자동 분석**되고, 앱 안에서 **즉석 촬영**도 됩니다.
> 5. **Windows에서 바로 실행**됩니다(`run_app.bat`). 휴대폰은 같은 와이파이로 접속해 그대로 테스트
>    가능하고, **정식 앱 연결용 REST API**(`run_api.bat`)도 만들어 뒀습니다.
>
> 사용법/적용/테스트/추가 데이터/모델 대안/정식앱 로드맵은 `개발자_전달.md`에, 원본 분석은
> `ANALYSIS.md`에 정리했습니다. 데이터 없이도 데모가 돌아가니 `run_app.bat`부터 눌러보시면 됩니다.
> Gemini 키만 넣으시면 LLM 설명도 붙습니다.

## 11. 향후 발전 제언 (무엇을 더하면 좋은가)

우선순위(★ 많을수록 효과·수요 큼) 기준입니다.

### 11-1. 기능
| 우선 | 기능 | 왜 |

| ★★★ | **필지(영역) 단위 분석** — 점 1개가 아니라 임야 경계 폴리곤을 받아 면적·평균입지 산정, **식재 본수·예상 탄소흡수량**까지 | 산주·공무원의 실제 의사결정 단위가 '필지'. 사업성 직결 |
| ★★★ | **현장조사 모드** — 다중 사진 일괄 업로드 → 좌표·추천·촬영시각 표 자동생성 → **CSV/GeoJSON/Shapefile 내보내기** | 산림공무원 대량 처리. 행정 연계 |
| ★★★ | **식재계획 리포트 PDF 자동생성** (수종·본수·시기·관리법·근거) | 결재·납품 산출물. 바로 사업화 가능 |
| ★★☆ | **피드백 루프** — 실제 식재/활착 결과를 사용자가 기록 → *진짜 라벨* 축적 → SDM 재학습 | 시간이 갈수록 똑똑해지는 구조. 데이터 자산화 |
| ★★☆ | **수종 상세 카드** — 묘목 단가·식재 적기·관리법·실물 사진, 국가표준식물목록/도감 연계 | 비전문가 신뢰·전환율↑ |
| ★★☆ | **병해충 경보 연계** — 소나무재선충 발생지 회피/경고 | 잘못된 추천 방지. 정책 정합 |
| ★☆☆ | 오프라인 캐시(산악 음영지대), 즐겨찾기/이력, 접근성(시니어 음성안내), SNS 공유카드 | 사용성·도달률 |

### 11-2. 데이터·모델
| 우선 | 항목 | 효과 |
|---|---|---|
| ★★★ | **토양도·기후격자 결합** — `SiteContext`에 필드 추가 + 요인함수 1개씩 연결 | 적지 정확도 핵심 도약(구조는 이미 확장 가능) |
| ★★★ | **탄소흡수량 정량화** — 수종별 생장·탄소계수(국립산림과학원) 적용 | "어떤 수종을 심으면 30년간 몇 tCO₂" — 배출권·정책 직결 |
| ★★☆ | **미래기후(SSP) 시나리오** — "지금 적합 vs 2050 적합 수종" 동시 제시 | 기후변화 적응 조림. 차별화 포인트 |
| ★★☆ | **SDM 앙상블 + 공간블록 CV + 불확실성 표시** | 모델 신뢰도·정직한 성능 평가 |
| ★☆☆ | 전국 임상도 확대, 표본점(국립수목원/GBIF) 보강 | 적용 범위·학습량 |

### 11-3. 엔지니어링·신뢰성
- **대용량 고속화**: 임상도를 **GeoParquet/FlatGeobuf**로 변환 + 지역 타일링(전국 확대 대비).
- **API 운영**: 호출 timeout·재시도·rate limit, Gemini 키는 Secret Manager, 결과 캐싱.
- **관측성·프라이버시**: 로깅/모니터링, 민감 필지 좌표 비공개 옵션.
- **품질 게이트**: GitHub Actions로 pytest 자동화, 타입힌트·정적검사(ruff/mypy).
- **검증 루프**: 산림청 적지적수 표준표 대조, 전문가 검수, 추천 근거 출처 표기·면책 고지.

### 11-4. 참고할 만한 유사 사례 (개념 참고용 — 코드 직수입은 비권장)
- **iTree / USFS Climate Change Tree Atlas** — 입지·기후 기반 수종 적합도·분포변화(접근 방식 참고).
- **biomod2(R), MaxEnt** — SDM 앙상블/presence-only의 사실상 표준(방법론 참고).
- **산림청 FGIS·산림빅데이터거래소, 국립산림과학원 수종분포변화 예측** — 국내 데이터·기준 정합.
> 스택(R/자바 등)·범위가 달라 코드 이식은 비효율적입니다. **방법론(앙상블·공간 CV·기후엔벨로프)**
> 만 차용하고, 구현은 현재의 파이썬 엔진을 확장하는 편이 낫습니다.
