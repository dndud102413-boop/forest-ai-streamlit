## 변경 1. "AI 모델"이 사실은 학습을 안 하고 있었습니다 (가장 중요)

### 기존이 한 일 (쉽게 풀면)
개발자는 먼저 **나무의 점수를 매기는 규칙을 손으로** 썼습니다. 예를 들어:

```python
def make_priority(row):           # ← 사람이 직접 짠 규칙
    score = 0
    if "5" in 영급 or "6" in 영급:  score += 2     # 나이 많으면 +2
    if "대경" in 경급:             score += 2     # 굵으면 +2
    if "밀" in 밀도:               score += 1     # 빽빽하면 +1
    if 고도 >= 500:                score += 2     # 높으면 +2
    ...
    if score >= 5: return "높음"
    elif score >= 3: return "보통"
    else: return "낮음"

# 이 규칙으로 '정답 라벨'을 만든 뒤
df["priority"] = df.apply(make_priority)

# 똑같은 입력으로 머신러닝을 '학습'
X = df[[영급, 경급, 밀도, 고도, ...]]   # 규칙이 쓴 바로 그 항목들
y = df["priority"]                      # 규칙이 만든 답
RandomForest.fit(X, y)                  # → 정확도 99%!
```

### 먼저 결론 한 줄
> 그 ML은 **고장 나서 안 돌아간 게 아닙니다. 잘 돌아갑니다 — 그런데 의미가 없습니다.**
> 정확도 99%가 나오지만, 그 99%는 "현장을 잘 맞혔다"가 아니라 **"내가 손으로 짠 규칙을
> 그대로 잘 베꼈다"** 는 뜻이라, 새로 배운 게 0입니다.

### 왜 의미가 없나 — '계산기' 비유 (가장 직관적)
> `y = 2x + 3` 이라는 **공식을 내가 이미 알고 있는데**, x를 잔뜩 넣어 y를 계산해 데이터를
> 만든 뒤, 그 (x, y)로 ML을 "학습"시키는 것과 같습니다. ML은 당연히 `2x+3`을 재발견해
> **정확도 100%** 를 냅니다. 그런데 나는 `2x+3`을 **처음부터 갖고 있었습니다.** ML은 내가
> 이미 아는 걸 다시 알아냈을 뿐이고, 그 100%는 "`2x+3`이 현실에서 옳은 공식인가"에 대해선
> **아무것도 증명하지 못합니다.**

위 노트북에서 ML이 한 일이 딱 이것입니다 — **사람이 쓴 if-else 규칙을 그대로 흉내 내기.**
그래서 ML을 통째로 지우고 규칙을 직접 써도 **결과가 똑같고, 더 빠르고, 더 투명합니다.**

### 당신이 헷갈린 부분을 콕 집어서
- **"학습데이터에 검증·테스트 데이터를 섞은 건가요?"** → 아닙니다. 그건 *다른 종류의 실수*
  (데이터 오염)입니다. 여기서는 **train/test를 깨끗하게 나눠도 문제가 그대로 남습니다.**
  왜냐하면 *정답(라벨) 자체가 입력으로부터 공식으로 계산*되기 때문에, 어떤 줄이든 "그 줄의
  답은 그 줄의 입력만 보면 공식으로 즉시 나오는" 구조라서요.
- **"과적합 문제인가요?"** → 아닙니다. 과적합은 모델이 *훈련 데이터의 잡음까지 외워서 새
  데이터에선 틀리는* 현상입니다. 여기선 정반대로 **테스트에서도 거의 100%로 잘 맞힙니다**
  (테스트 답도 같은 공식이 만들었으니 '새 정보'가 아니거든요). 정확한 이름은 **순환 학습/
  자기참조(데이터 누수)** 입니다 — *답을 만든 공식으로 그 답을 다시 맞히는* 셈.
- **"규칙을 베꼈는데 왜 ML이 제대로 안 도나요?"** → ML은 **제대로 돕니다.** 안 도는 게
  아니라 *해봐야 얻는 게 없는* 겁니다. 오히려 높은 정확도가 "똑똑한 걸 만들었다"고
  **착각하게 만드는** 점이 더 위험합니다.

### 그럼 규칙과 ML은 각각 무슨 역할이어야 하나
| | 잘못된 설계(기존) | 올바른 설계 |
|---|---|---|
| **규칙의 역할** | 답을 *직접 만드는* 출제자 겸 정답표 | 사람이 **이미 아는 명확한 논리**를 투명하게 적용 |
| **ML의 역할** | 그 규칙을 *베끼는* 복사기(쓸모 0) | 사람이 **모르는 패턴**을 *실제 데이터*에서 학습 |
| **정답(라벨)의 출처** | 규칙(사람이 지어낸 답) | **현실 관측**(실제로 일어난 일) |

> 핵심 원칙: **"답을 만든 것"과 "답을 학습하는 것"이 같은 규칙을 공유하면 안 됩니다.**
> ML이 진짜 가치를 내려면 *정답이 규칙이 아니라 현실에서* 와야 합니다.

### 그래서 어떻게 바꿨나 (※ ML을 버린 게 아니라 *교체* 했습니다)
> 개발자는 ML을 쓰고 싶어 했고, 그건 좋은 방향입니다. 문제는 'ML을 쓴 것'이 아니라
> '**잘못된 방식**으로 쓴 것'이었습니다. 그래서 **잘못된 ML만 걷어내고, 올바른 ML을
> 추천에 기본으로 넣었습니다.**

1. **잘못된 ML(순환학습)만 제거.** RandomForest/XGBoost로 규칙을 베끼던 부분만 삭제.
2. **올바른 ML(SDM)을 추천에 기본 사용.** 임상도에 기록된 **"그 자리에 실제로 자라는
   수종"** 을 정답으로 삼아 환경→수종 패턴을 학습합니다(변경 6, 기본 ON). 게다가 **모델의
   실측 신뢰도(f1)에 비례해 영향력을 자동 조절**합니다 — 데이터가 좋으면 ML이 강하게,
   부족하면 약하게 반영해 *언제나 안전*합니다.
3. **여기에 '투명한 적지적수 점수'를 함께(하이브리드).** ML 확률 + 지식기반 점수를 섞어,
   *데이터로 배우면서도* "이 위치는 온대북부 + 고도 1200m + 북사면 → 잣나무 적합" 처럼
   **왜 이 수종인지 근거도 보이게** 했습니다. (ML만 쓰면 데이터가 적을 때 약하고 설명이
   안 되며, 규칙만 쓰면 데이터에서 못 배웁니다 → 둘을 합친 게 가장 우수)
4. **규칙 기반 '현재 숲 진단'은 유지.** 진단은 원래 규칙이 적합한 영역이라 그대로 둡니다(변경 2).

## 변경 2. '진단'은 유지하고, '추천'을 새로 더했습니다 (둘 다 제공)

### 기존
`forest.contains(point)` 로 **그 자리에 이미 있는 임분**(임종/수종/경급/영급/밀도)을
조회해 보여주는 게 전부였습니다. 즉 *"여기엔 30년생 소나무가 있네요"* 까지. (원래 목표인
"**뭘 심으면 좋을지**"는 빠져 있었습니다.)

### 바뀐 것 — 두 기능을 모두 제공
1. **🩺 현재 숲 진단(유지)**: 그 자리에 이미 자라는 숲의 *생육 적합도·병해충 취약도·관리
   우선순위* 를 **투명한 규칙**으로 평가합니다. (원본의 규칙을 정리해 되살렸고, 변경 1에서
   말한 대로 ML로 감싸지 않습니다.)
2. **🌱 식재 수종 추천(신설·핵심)**: 입지 환경(기후대·고도·경사·향)을 보고 *"여기엔 이런
   나무를 심으면 좋습니다"* 를 순위·근거로 냅니다.

> 즉 **"지금 뭐가 있나(진단) + 새로 뭘 심으면 좋나(추천)"** 를 한 화면에서 같이 봅니다.

추천 예시:

```
🌱 추천 수종
1. 신갈나무  92.4점  (기후대 적지 / 고도 1233m가 적정범위 / 경사 적합)
   ⚠ 병해충 유의: 참나무시들음병
2. 잣나무   84.8점  ...
```

이게 **이 프로젝트의 핵심 신규 기능**입니다([recommender.py](forest_reco/recommender.py)).

## 변경 3. "사진의 점이 격자 안에 들어가는지" — 벡터 vs 래스터 정리

개발자가 걱정한 부분입니다. 핵심은 **데이터가 두 종류**라는 것:

| | 임상도(.shp) | DEM(.tif) |
|---|---|---|
| 형태 | **벡터** — 임의 모양 다각형(숲 경계) | **래스터** — 바둑판 격자(픽셀) |
| 비유 | 지도 위에 그린 **나라 국경선** | 모눈종이에 칠한 **고도 색칠** |
| 점이 들어가는지 | "어느 **나라 안**인가" (다각형 포함) | "어느 **모눈칸**인가" (격자 인덱스) |
| 방법 | `point.within(polygon)` | `dataset.index(x,y) → 행,열` |

> 즉 **임상도는 격자가 아니라서 격자 계산이 필요 없고**(다각형 포함 판정), **격자 계산이
> 필요한 건 DEM 쪽**입니다. 개발자의 직관은 'DEM'에 대해선 정확했습니다.

### 기존의 함정과 수정
- ❌ DEM은 EPSG:5179(미터 좌표)로 바꿔놨는데, 거기에 **위경도(도 단위)를 그대로** 넣어
  엉뚱한 칸을 읽었습니다.
  > 비유: 미터자로 그린 지도에 "위도 37도" 같은 **다른 단위 눈금**을 들이댄 격.
- ❌ 격자 밖 좌표가 **음수 칸 번호**를 만들면, 컴퓨터가 조용히 **반대편 끝 칸**을 읽어
  "그럴듯한 가짜 고도"를 정답처럼 돌려줬습니다.
  > 비유: 책 -3쪽을 펼치라 하면 뒤에서 3번째 쪽이 펴지는데, 그걸 모르고 그 내용을 믿는 것.
- ✅ 수정: [`geo.sample_raster()`](forest_reco/geo.py)가 ① 좌표를 항상 데이터 단위로 변환
  ② 격자 범위를 벗어나면 "값 없음"이라고 정직하게 알림 ③ 빈 칸(nodata) 처리까지 합니다.

## 변경 4. 아예 실행이 안 되던 버그들

기존 노트북은 **그대로 돌리면 에러로 멈추는** 부분이 있었습니다.
- `aspect_to_direction()`(경사 방향을 '남향/북향'으로 바꾸는 함수)을 **호출은 하는데 정의가
  없어서** 멈춤 → 만들어 넣음.
- 일부 셀은 **들여쓰기 오류(문법 에러)** → 모듈화로 제거.
> 비유: 요리법에 "②번 양념을 넣으세요"라고 적혀 있는데 정작 ②번 양념 만드는 법이
> 빠져 있던 것. 그 단계를 채워 넣었습니다.

## 변경 5. "여기저기 흩어진 한 번 쓰고 버린 코드"를 정리

기존 노트북은 121칸 중 상당수가 streamlit 실행/종료/터널을 **수십 번 반복 시도한 흔적**
이었고, 데이터를 매번 다시 읽어 느렸습니다.

- ✅ 핵심 로직을 **재사용 가능한 부품(모듈)** 로 분리 → 앱·API·노트북이 같은 엔진을 공유.

## 마무리 — 한 문장으로
> 기존은 **좋은 재료(임상도·DEM·Gemini)** 를 가졌지만 **(1) 학습을 안 하는 AI, (2) 목표와
> 다른 기능, (3) 꼬인 좌표계, (4) 멈추는 코드** 라는 문제가 있었고, 이번 작업은 그 넷을 모두
