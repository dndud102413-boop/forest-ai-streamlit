# 코드 재진단 리포트 (실데이터 견고성·엣지케이스·확장성)

> 요청: "현재 수준에서 코드가 정상 작동할지, 미세한 오류·에러는 없을지, 입력 데이터
> 형태가 제대로 가정돼 있는지, 확장성이 보장되는지" 재검토.
> 방법: 다중 에이전트 적대적 재진단(파일 직접 실행·재현 검증) → **확정 7건** 도출 →
> 6건 코드 반영 + 1건 로드맵. 반영 후 **테스트 53개 통과**.

---

## 종합 결론

- **정상 작동합니다.** 합성데이터 end-to-end 구동 + Windows 실구동(스크린샷) + 자동 테스트 53개 통과.
- **실데이터를 넣었을 때의 잠재 문제 7건**을 미리 찾아 대부분 코드로 막았습니다. 핵심은
  "크래시"가 아니라 **"조용히 틀린 결과를 내는"** 유형이라, 안 잡았으면 현장에서 신뢰를
  잃기 쉬운 것들이었습니다.

---

## 발견·조치 내역 (7건)

| # | 심각도 | 문제 (어떤 입력에서) | 조치 |
|---|---|---|---|
| 1 | **치명** | 임상도 `.prj`가 없고 실제가 **EPSG:5181**인데 5179로 조용히 가정 → 좌표 1000km+ 어긋나 **모든 질의가 '데이터 없음'처럼 무음 실패** | ✅ `.prj` 없으면 **경고 + 좌표범위 정합성 검사로 즉시 에러**(무음 실패 차단) |
| 2 | 높음 | `KOFTR_NM`이 **'참나무류'·'기타활엽수'·'Pinus densiflora'** 등이면 KB 매칭 실패 → 경험가점/SDM 신호 **무음 탈락** | ✅ 학명 토큰 추가 + **`match_all()` 그룹 팬아웃**(참나무류→Quercus 멤버 분배), 미매칭은 분모에 남겨 희석 |
| 3 | 높음 | DEM에 좌표계가 없으면 지리좌표(도)를 투영(미터)으로 오인 → **경사/향이 비현실적으로 왜곡** | ✅ CRS 없으면 **해상도로 4326/5179 추정 + 경고**, 재투영 경로로 정확 계산 |
| 4 | 높음 | **평지(경사 0)** 가 항상 **'북향'** 으로 잘못 인코딩 | ✅ 평지 향을 **NaN→'평지'** 로 마스킹 |
| 5 | 높음 | 추천 요인이 하드코딩 리스트라 **토양/기후 레이어 추가가 구조적으로 어려움** | ✅ **요인 레지스트리 + 가중치 자동정규화**로 리팩토링. 배수(토양) 요인 추가를 예시로 구현(`soil_drainage`) |
| 6 | 높음 | 수종명 매칭이 수작업 별칭에 의존 → 전국 데이터에서 **대량 미매칭** | ✅ #2와 함께 처리(학명·그룹라벨·분모희석). KB 전국 확장·별칭 CSV화는 로드맵 |
| 7 | 높음 | 데이터 소스가 **특정 파일명·단일 타일** 전제 → 전국 타일링/소스 교체 어려움 | 🔜 **로드맵**: Provider 추상화 + DEM windowed read(아래 §확장성) |

> 1~5는 **코드 반영 완료**, 6은 핵심부 반영(전국 KB 확장은 데이터 작업), 7은 설계 로드맵.

## "입력 데이터 형태"는 제대로 가정돼 있나

실제 임상도/DEM이 가질 수 있는 변형에 대한 현재 내성:

| 입력 변형 | 현재 동작 |
|---|---|
| 임상도 CRS 5179 / 5181 | `.prj`에서 자동 인식. `.prj` 없으면 경고+정합성 검사로 **오라벨 차단**(#1) |
| `FRTP_NM` = 침/활/혼 **또는** 인공/천연림 | 안전(추천 점수에 미사용, 표시용) |
| `KOFTR_NM` = 단일종/그룹라벨/학명병기/지방형 | `match_all` 팬아웃 + 학명·별칭으로 대부분 흡수(#2) |
| `AGCLS_NM` = '3영급'/'Ⅲ'/정수, `DMCLS/DNST` 변형 | 안전(추천에 미사용, 표시용) — 부분문자열 매칭 제거로 오분류 없음 |
| 누락 컬럼 | `row[col] if col in row.index else None` 가드 |
| MultiPolygon / 빈 geometry | `representative_point`·`within`으로 처리(빈 geometry는 사전 `inspect` 권장) |
| DEM 4326 / 5179 / CRS 없음 / nodata 상이 / 멀티밴드 | 좌표변환·재투영·nodata·밴드1 처리(#3 포함) |

> **권장 절차**: 실데이터 받으면 먼저
> `python scripts\prepare_data.py inspect --shp ... --dem ...` 로 CRS·고유값·범위를
> 확인하세요. `KOFTR_NM` 고유값에 그룹라벨이 많으면 `match_all` 규칙(참나무류 등)에
> 맞는지 한 번 점검하면 됩니다.

## 확장성은 보장되나

- ✅ **새 입지 요인 추가가 1줄**: `SiteContext`에 필드 추가 + 요인함수 1개 작성 +
  `factor_fns`에 등록. 가중치는 **활성 요인 합으로 자동 정규화**되어 기존 점수 균형을
  깨지 않습니다. (배수 요인을 이 방식으로 실제 추가해 시연 — [recommender.py](forest_reco/recommender.py))
- ✅ **새 수종/목적 추가**: `species_db.json`에 항목 추가 / `GOALS`에 매핑 추가.
- ✅ **LLM·앱·API가 같은 엔진 공유**: 추천 로직은 한 곳, 표현부만 다름.
- 🔜 **전국 확대·소스 교체(로드맵)**: 데이터 적재를 `Provider`(query만 노출) 인터페이스로
  추상화하고, ① 파일 ② 타일/공간DB 구현을 분리. DEM은 점 주변만 **windowed read**로
  국소 계산해 전영역 재투영을 제거하면 전국 스케일에서도 빠릅니다. (현재 지역 단위는 전체
  적재로 충분)

## 남은 권고 (운영 전 체크)
- 실데이터 1필지로 `inspect` → `run_demo` 한 번 돌려 좌표·고도·추천이 상식적인지 눈으로 확인.
- `KOFTR_NM` 미매칭 비율이 높으면 KB/별칭 보강(데이터 작업).
- Gemini 호출에 timeout·재시도(운영 안정성), 키는 환경변수/시크릿으로.
